Test result ingestion API

Feature PRD — POST /api/v1/test-results

Status Draft
Author Colin Steele
Date 2026-03-12
Phase v1 — post-MVP

Problem

The core traceability graph connects user needs → requirements → implementation artifacts (code, schematics, CAD). What it cannot yet do is close the evidentiary loop: show that a requirement was not just implemented, but verified.

Test results currently live in a fragmented world — CI/CD pipelines, factory floor ATE systems, hand-rolled ATP scripts, bench test logs — each with its own format and no standard bridge to the traceability layer. Connecting them to the graph today requires manual effort: someone has to look at a test result, find the requirement it validates, and update a spreadsheet or document to record the relationship. That work is tedious, error-prone, and almost always behind the current state of the codebase.

Auditors and certification reviewers routinely ask one question above all others: "Can you show me that this requirement is actually implemented and verified?" The gap between test outputs and the traceability model is exactly where teams fail to answer that question cleanly.

Goal

Provide a well-defined HTTP endpoint that any tooling — CI pipelines, ATE adapters, hand-rolled scripts, factory floor software — can POST to in order to inject test results into the graph as first-class nodes. Once ingested, a result node is connected to its test case node via a single edge. The graph resolves all downstream relationships (requirements, implementation refs, design objects) from there. Callers do not need to understand or replicate the traceability model.

Design principle: push the heterogeneity problem to the people who own it. Customers write a thin adapter that normalizes their tool's output to this schema. The system does the rest.

Non-goals

  • The system does not parse arbitrary test output formats natively. There is no built-in support for JUnit XML, pytest JSON, LabVIEW TDMS, or any other tool-specific format. Customers provide adapters.
  • This endpoint does not replace or replicate requirement IDs. The caller supplies only a test_case_ref. Requirement resolution is the graph's job.
  • Real-time streaming of test output is out of scope. Results are pushed as complete execution payloads after a run finishes.

User stories

CI/CD engineer

"After my test suite runs in GitHub Actions, I want to push the results to the traceability system with a single curl call so that the audit trail is updated automatically without anyone touching a spreadsheet."

Factory floor operator

"When our ATE finishes a unit test, I want the pass/fail status and all measured values to flow into the traceability graph so the DHF is current without manual data entry."

Quality engineer

"I want to query the graph and see, for any given requirement, the full history of test executions that have verified it — including who ran them, when, on what environment, and whether they passed."

Auditor / regulator

"I want to trace from a user need to the lines of code that implement it, then forward to the test case that verifies it, then to the actual execution evidence — date, result, measured values — without being handed a spreadsheet and told to trust it."

Schema

Endpoint

POST /api/v1/test-results — Content-Type: application/json

Top-level fields

FieldRequiredTypeDescription
execution_idrequiredstringUnique ID for this run. UUID, CI build ID, or any stable string. Used to correlate all test case results from a single execution.
executed_atrequiredISO 8601Timestamp of when the run completed. e.g. 2026-03-12T14:32:00Z
statusrequiredenumRollup status. One of: passed failed partial skipped error
test_casesrequiredarrayArray of individual test case results. At least one entry required.
executoroptionalobjectWho or what ran the tests. Fields: name, type (one of human ci ate script), environment.
sourceoptionalobjectProvenance metadata. Fields: tool, version, raw_log_url. Used for audit trail linking back to original output.

test_cases[ ] — each item

Key design decision: test_case_ref is the only required reference. The system walks the graph from this edge to find requirements, implementation artifacts, and design objects. Callers never need to supply requirement IDs — that would force them to understand the traceability model, which is exactly the friction we're eliminating.

FieldRequiredTypeDescription
result_idrequiredstringUnique ID for this result node. UUID preferred. Becomes a first-class node in the graph with its own edges.
test_case_refrequiredstringID of the test case node this result is an execution of. Must match an existing node. This single edge is how requirements, design objects, and implementation refs are resolved.
statusrequiredenumOne of: passed failed skipped error
duration_msoptionalintegerExecution time in milliseconds.
notesoptionalstringFree text. Failure reason, deviation notes, operator observations. No structure enforced.
measurementsoptionalarrayMeasured values vs. limits. Designed for ATE and hardware bench tests. See below. Software tests typically omit this entirely.
attachmentsoptionalarrayLinked artifacts — scope captures, log files, screenshots. Each item: name, url, mime_type.

measurements[ ] — each item

This array exists specifically to make ATE and hardware test results first-class citizens. Rather than embedding pass/fail as a blob in notes, measured values and their limits are structured — making them queryable and auditable independently of the caller's assertion.

FieldRequiredTypeDescription
namerequiredstringParameter name. e.g. supply_voltage_3v3, rise_time_us
valuerequirednumberActual measured value.
unitoptionalstringUnit of measure. e.g. V, mA, us, dBm
limit_lowoptionalnumberLower bound of passing range.
limit_highoptionalnumberUpper bound of passing range.
passedoptionalbooleanCaller's assertion of in-limit status. The system stores both the assertion and the raw value so auditors can verify limits independently.

Customer adapter responsibility

This API deliberately does not parse native tool formats. The expectation is:

  • Customer writes a thin script or CI step that reads their tool's output (JUnit XML, pytest JSON, LabVIEW TDMS, custom ATP log, etc.) and maps it to this schema.
  • The adapter POSTs the normalized payload to the local API endpoint.
  • The system handles graph insertion, edge resolution, and audit trail.

This design keeps the core clean and puts the heterogeneity problem where it belongs — with the teams who own those tools and formats. The schema is intentionally minimal so that writing an adapter is a small, one-time task, not an ongoing integration burden.

Graph behavior on ingest

On receipt of a valid payload, the system:

  • Creates one result node per entry in test_cases[], with all supplied fields as node properties.
  • Creates an edge from each result node to its test_case_ref node.
  • Resolves and surfaces (but does not duplicate) the downstream edges already present on the test case node: links to requirements, implementation artifacts, design objects.
  • If measurements[] is present, stores each measurement as a sub-object on the result node, queryable independently.
  • If test_case_ref does not match an existing node, returns a 422 with a descriptive error. Results are not partially inserted — the entire payload is atomic.

Open questions

  • Should we support idempotent re-submission by execution_id? (i.e., POSTing the same execution twice updates rather than duplicates.) Likely yes — CI pipelines retry.
  • Should result_id be caller-supplied or server-generated? Caller-supplied allows the adapter to be deterministic and idempotent. Server-generated is simpler but breaks re-submission.
  • Do we need a bulk endpoint (POST /api/v1/test-results/batch) for large ATE runs with hundreds of test cases per execution? Or is the array in the single endpoint sufficient?
  • What is the right behavior when test_case_ref doesn't exist yet — hard reject, or queue for later resolution? The "backwards workflow" reality (code first, test cases formalized later) suggests soft-queuing may be valuable.

Out of scope for v1 / future consideration

  • Native parsers for common formats (JUnit, pytest, LabVIEW). Could be shipped as optional adapter libraries.
  • Webhook / push notifications when a result causes a requirement's verification status to change.
  • Dashboard views aggregating test result trends over time per requirement.
  • Support for ad-hoc results that do not map to an existing test case node (would require inline test case creation in the same payload).
